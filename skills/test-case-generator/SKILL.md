## 元能力：环境识别与适配

在开始任何测试工作前，先识别目标环境。

### 第一步：识别语言和测试框架

扫描项目根目录，自动检测语言和测试框架。

| 指标文件 | 语言 | 检测测试框架 |
|----------|------|-------------|
| `go.mod` | Go | `testing` + testify(可选) |
| `package.json` | JS/TS | jest/vitest/mocha |
| `pyproject.toml` / `requirements.txt` | Python | pytest/unittest |
| `pom.xml` / `build.gradle` | Java | JUnit/TestNG |
| `*.csproj` | C# | NUnit/xUnit |
| `Cargo.toml` | Rust | 内置 `#[test]` |

详细识别策略和适配规则见 [测试发现与盘点参考](references/test-discovery.md)。

### 第二步：识别测试源

测试用例可能存在于多种位置：

| 测试源 | 识别方法 | 典型场景 |
|--------|----------|----------|
| Git 仓库代码 | 搜索测试文件模式 (`*_test.go`, `test_*.py`) | 单元测试、集成测试 |
| Git 仓库配置 | 搜索 JSON/YAML 用例文件 | 数据驱动测试 |
| 飞书云文档 | `feishu-mcp` 工具搜索"测试计划""测试用例" | 手工测试、验收测试 |
| 飞书多维表格 | `feishu-mcp` 工具定位表格 | 测试管理、执行追踪 |
| Excel/Word 文件 | 本地文件搜索 | 离线测试管理 |
| Jira/Confluence | API 或导出 | Issue 关联测试 |
| 自研系统 | 具体分析 | 定制化测试流程 |

**注意**：不要假设测试只在代码中。很多项目的测试用例分散在多个系统中。

### 第三步：适配输出格式

根据识别结果适配：
- **命名规范** — 使用目标语言惯例（Go: `TestXxx`, Python: `test_xxx`, JS: `xxx.test.ts`）
- **断言库** — 使用项目已有依赖（不引入新库）
- **目录结构** — 遵循项目约定
- **配置文件** — 如需要则按框架生成（pytest.ini, jest.config.ts 等）

## 能力一：通过测试理解代码

### 适用场景

- 刚接触一个新仓库或新模块
- 代码文档不足，需要通过实验理解行为
- Review 他人代码时需要验证自己的理解
- 任意语言的陌生代码

### 工作流程

```
读代码入口 → 猜测行为 → 写测试验证 → 调整理解 → 整理发现
```

**关键策略**：将每个不确定的行为假设转化为一个测试。不追求完美断言，先用 `t.Logf` / `print` / `console.log` 记录实际输出，观察后再确定断言。

```go
// 探索性测试：不确定对空输入的处理方式
func TestExplore_EmptyInput(t *testing.T) {
    result, err := ProcessData(nil)
    t.Logf("result=%v, err=%v", result, err) // 先观察
}
```

### 协作要点

- 主动向人类报告"未预期的行为"，请确认是 bug 还是设计意图
- 对 subagent，将发现写回任务报告而非只返回代码
- 探索性测试标注 `// 探索性测试` 与正式测试区分

## 能力二：通过测试定位 Bug

### 适用场景

- 代码行为异常但原因不明
- 偶发 bug 难以稳定复现
- 怀疑是某个模块的问题但需确认

### 工作流程

```
稳定复现 → 逐层缩小范围 → 隔离最小复现 → 确认修复
```

**关键策略**：二分法逐层排除。每次测试注释中记录推理过程。

```
全链路测试（失败）
  → 测试模块 A（通过）→ 排除 A
  → 测试模块 B（失败）→ 问题在 B
    → 测试 B.b1（通过）
    → 测试 B.b2（失败）→ 问题在 b2
```

### 协作要点

- 每缩小一层范围，向人类汇报进展
- 找到疑似根因时先请确认再动手修复
- 对 subagent，使用并行 Agent 同时探索多个可疑模块

## 能力三：系统化测试设计

### 适用场景

- 新功能需要完整测试覆盖
- 已有项目需要补充测试
- 提升测试覆盖率

### 工作流程

```
分析被测目标 → 设计测试策略 → 生成用例清单 → 执行/分配 → 验证
```

#### 测试策略分层

| 复杂度 | 特征 | 策略 |
|--------|------|------|
| 简单函数 | 纯逻辑、无副作用 | 表驱动测试覆盖所有分支 |
| 有依赖的函数 | 需要外部数据/服务 | Mock 依赖 + 边界测试 |
| 模块级交互 | 多组件协作 | 集成测试 + 真实依赖 |
| 用户功能 | 端到端流程 | 冒烟测试 + 关键路径 |
| 产品功能 | 人类可操作的功能 | 手工测试 + 验收标准 |
| 合规/安全 | 必须满足的要求 | 检查清单 + 合规矩阵 |

#### 风险优先排序

1. **P0** — 核心业务路径（失败影响最大）
2. **P1** — 边界条件（最容易出 bug）
3. **P2** — 异常处理（防御性编程）
4. **P3** — 性能和兼容性

#### 非代码测试设计

本技能不只适用于代码测试。对产品功能、操作手册、合规要求等：

| 测试目标 | 输出格式 | 示例 |
|----------|----------|------|
| 产品功能 | 手工测试步骤 | Markdown 文档，含前置条件/操作/预期 |
| 操作手册 | 验证检查清单 | 每个操作步骤的预期结果 |
| API 接口 | 接口测试 | HTTP 请求/响应断言 |
| 合规要求 | 合规检查项 | 法规条款 vs 系统功能对照 |
| 安全测试 | 安全检查清单 | OWASP Top 10 检查项 |
| 性能测试 | 性能基准 | 指标 + 阈值 + 测试方法 |

## 能力四：测试盘点与评估

### 适用场景

- 用户说"帮我看看现在有哪些测试"
- "评估一下测试覆盖情况"
- "测试质量怎么样"
- 需要向团队汇报测试状态

### 工作流程

```
扫描所有测试源 → 分类统计 → 覆盖缺口分析 → 质量评估 → 输出报告
```

#### 第一步：全源扫描

不要只看代码。同时扫描：
- Git 仓库中的测试文件和配置
- 飞书文档中的测试计划和用例
- 飞书多维表格中的测试管理数据
- 其他系统中的测试资产

使用对应的工具访问：
- Git → `Glob`/`Grep` 搜索测试文件
- 飞书文档 → `feishu-mcp` 工具
- 本地文件 → `Read`/`Glob`

#### 第二步：分类统计

按以下维度分类：
- **按类型**：单元/集成/端到端/手工/自动化
- **按优先级**：P0/P1/P2/P3
- **按状态**：通过/失败/跳过/未执行
- **按所有者**：开发/QA/自动化
- **按时效**：最近更新/长期未维护

#### 第三步：覆盖缺口分析

对照源代码/需求文档，识别：
- 完全未覆盖的模块
- 覆盖但不充分的（只有正向、缺少边界/异常）
- 已有但质量低（欺骗性测试、flaky 测试）

#### 第四步：输出盘点报告

详细报告模板见 [测试发现与盘点参考](references/test-discovery.md)。

### 协作要点

- 盘点结果以 Markdown 表格形式呈现，便于人类快速扫描
- 覆盖缺口按风险优先级排序
- 每个缺口附带具体的补充建议

## 能力五：测试优化与迭代指导

### 适用场景

- 用户说"测试质量不高，怎么改进"
- "有很多 flaky 测试"
- "测试维护成本太高"
- 需要制定长期测试改进计划

### 常见问题与策略

| 问题 | 症状 | 策略 |
|------|------|------|
| 覆盖率不足 | 核心 bug 无测试拦截 | P0 功能优先 → 分层推进 |
| 测试质量低 | 通过但 bug 仍出 | 检查断言有效性 → 补充边界 |
| Flaky 测试 | 偶尔失败 | 隔离 → 消除非确定性 → 降级手工 |
| 维护成本高 | 改代码改大量测试 | 减少 Mock 耦合 → 重构测试结构 |
| 测试冗余 | 多个测试验证相同逻辑 | 合并去重 → 表驱动整合 |
| 测试过时 | 与当前代码不匹配 | 清理或更新 → 标记废弃 |

### 优化优先级

```
1. P0 功能无测试        → 立即（高风险）
2. Bug 回归测试缺失     → 立即（已知风险）
3. 核心逻辑覆盖不足     → 短期（1-2 周）
4. Flaky 测试修复       → 短期（影响信任）
5. 测试结构优化         → 中期（降低维护成本）
6. 覆盖率提升           → 长期（持续推进）
```

### 迭代节奏

每次优化迭代回答四个问题：
1. 上一轮补充的测试有效性如何？
2. 现在最大的覆盖缺口在哪？
3. 有没有新增的 flaky 测试？
4. 下一步优先做什么？

## 协作模式

### 模式 A：AI 主导（自动化）

适合模板明确的测试。直接读取代码 → 生成测试 → 验证。

### 模式 B：人机协作（推荐）

1. 分析代码，输出**测试用例清单**（不写代码）
2. 用户审核清单，调整优先级
3. 按确认的清单逐一生成
4. 每批暂停让用户验证

**输出示例**：
```
## FilterSheet 函数测试清单

| # | 场景 | 优先级 | 类型 |
|---|------|--------|------|
| 1 | 标准格式正确解析 | P0 | 正向 |
| 2 | 无分隔符返回错误 | P0 | 异常 |
| 3 | 空字符串处理 | P1 | 边界 |

请确认清单，我再开始编写。
```

### 模式 C：指导 subagent

向 subagent 的 prompt 中包含：
1. **测试目标** — 测什么、为什么
2. **策略指引** — 类型、关注点
3. **模板路径** — `references/` 中的模板
4. **约束条件** — 命名、断言、禁止事项
5. **期望输出** — 文件路径 + 覆盖场景

### 模式 D：外部系统集成

当测试用例存储在外部系统时：

**飞书文档**：
- 使用 `feishu-mcp` 的 `search-doc` 搜索测试文档
- 使用 `fetch-doc` 获取内容
- 解析文档中的测试用例结构

**飞书多维表格**：
- 使用 `list-docs` 定位表格
- 读取表格结构和数据
- 分析测试状态和覆盖

**其他系统**：
- 先了解可用的访问方式（API/导出/MCP）
- 选择最有效的集成策略

## 测试设计原则

### 等价类划分

每个输入参数识别：有效输入、无效输入、边界值。每类至少一个测试。

### 命名规范

| 测试类型 | 命名模式 |
|----------|----------|
| 单元测试 | `Test<功能>_<场景>` (Go) / `test_<功能>_<场景>` (Python) |
| 回归测试 | `TestRegression_<Bug描述>` |
| 探索测试 | `TestExplore_<假设>` |
| 调试测试 | `TestDebug_<问题>` |

### 断言规范

- 使用项目已有的断言库
- 禁止欺骗性测试
- 禁止忽略错误
- 禁止测试间依赖

## 扩展机制

本技能采用"核心方法论 + 可插拔模板"架构。当遇到能力不足的场景时，按以下机制扩展。

### 何时需要扩展

出现以下任一信号时，意味着技能需要补充：

| 信号 | 说明 |
|------|------|
| AI 反复问同样的项目特有问题 | 项目约定未被捕获，需要补充到项目文档或本技能 |
| 生成的测试不符合项目惯例 | 模板不够精确，需要补充项目特有模板 |
| 某类测试从未成功生成过 | 缺少该类型的知识，需要补充 reference |
| AI 无法识别项目的测试框架 | 检测规则不全，需要补充到 test-discovery.md |
| 用户说"每次都要手动改" | 自动化不足，需要捕获为模板或规则 |

### 扩展路径

根据不足的类型选择对应路径：

```
能力不足
├── 语言/框架不支持      → 扩展 references/test-discovery.md
├── 测试模板不匹配        → 扩展 references/ 下的模板文件，或新增
├── 项目特有约定未捕获    → 扩展项目 CLAUDE.md（不扩展本技能）
├── 外部测试源无法访问    → 扩展 references/test-discovery.md 的源扫描策略
└── 方法论缺失            → 扩展 SKILL.md 本体（新增能力章节）
```

#### 路径一：补充语言/框架支持

编辑 `references/test-discovery.md`，在对应表格中添加新条目：

```markdown
<!-- 在"检测测试框架"表格中添加新行 -->
| Kotlin | `kotest` in build.gradle.kts | Kotest |
```

**扩展要点**：
- 识别指标文件（什么文件说明用了这个框架）
- 测试文件命名模式（`*Test.kt`, `*Spec.kt`）
- 典型目录结构（`src/test/kotlin/`）
- 断言库惯例（`shouldBe`, `shouldEqual` 等）

#### 路径二：补充测试模板

当现有模板不匹配新项目/新框架时：

**选项 A — 扩展现有文件**：在 `references/go-templates.md` 等文件中添加新章节

**选项 B — 新建模板文件**：创建 `references/<framework>-templates.md`

```markdown
# <框架名> 测试模板参考

## 目录
- [单元测试模板](#单元测试模板)
- ...

## 单元测试模板

（框架特有的代码模板和约定）
```

新建文件后，在 SKILL.md 的"参考文件"表格中添加引用。

#### 路径三：捕获项目特有约定

**不要扩展本技能**，而是写入项目的 `CLAUDE.md`：

```
# 在项目的 CLAUDE.md 中添加
## 测试约定
- 测试文件放在 tests/ 目录而非与源文件同目录
- 使用 Given-When-Then 注释风格
- Mock 使用 gomock 而非 testify/mock
```

**为什么**：项目约定是项目级知识，不是通用方法论。本技能在运行时会读取项目 CLAUDE.md 获取约定。

#### 路径四：补充外部测试源集成

编辑 `references/test-discovery.md` 的"源类型识别"表格：

```markdown
| TestRail | API (需要 API Key) | 专业测试管理 |
```

同时补充扫描方法：
- 如何认证
- 如何获取用例列表
- 如何解析数据结构

#### 路径五：补充方法论

当遇到全新的测试目的或策略时，直接编辑 SKILL.md 添加新能力章节：

```markdown
## 能力六：<新能力名称>

### 适用场景
- ...

### 工作流程
- ...

### 协作要点
- ...
```

### 扩展时的注意事项

1. **保持分层清晰** — 方法论放 SKILL.md，模板放 references/，项目约定放项目 CLAUDE.md
2. **模板优于规则** — 提供具体的代码模板比写抽象规则更有效
3. **增量补充** — 发现一个缺口补一个，不要试图一次覆盖所有场景
4. **验证扩展** — 补充后用实际场景测试，确认 AI 能正确使用新内容

### 引导用户扩展

当 AI 意识到自身能力不足时，主动告知用户：

```
我注意到这个项目使用的是 <框架>，但我的测试模板中没有覆盖它。

建议扩展方式：
1. 在 references/ 中添加 <框架>-templates.md
2. 我可以先分析项目中的已有测试提取模式，帮你生成模板

要我现在分析项目的已有测试来生成模板吗？
```

### 自我改进循环

```
使用技能 → 遇到不足 → 记录缺口 → 补充扩展 → 下次生效
```

每次遇到能力不足的场景：
1. 记录具体是什么不足（语言？模板？方法论？）
2. 按扩展路径选择补充方式
3. 补充后在实际场景中验证
4. 将验证结果反馈给用户
