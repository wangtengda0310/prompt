## 语言/框架自动识别

### 识别策略

进入任意项目后，按以下顺序检测：

#### 第一步：检测语言

| 指标文件 | 语言 |
|----------|------|
| `go.mod` | Go |
| `package.json` | JavaScript / TypeScript |
| `requirements.txt` / `pyproject.toml` / `setup.py` | Python |
| `pom.xml` / `build.gradle` | Java |
| `*.csproj` / `*.sln` | C# |
| `Cargo.toml` | Rust |
| `Gemfile` | Ruby |
| `go.work` | Go (多模块工作区) |

#### 第二步：检测测试框架

| 语言 | 检测文件/依赖 | 测试框架 |
|------|--------------|----------|
| Go | 内置 `testing` 包 | `go test` |
| Go | `github.com/stretchr/testify` | testify |
| Go | `// +build integration` / `//go:build integration` | 集成测试标签 |
| Python | `pytest` in requirements | pytest |
| Python | `unittest` in imports | unittest |
| Python | `robot` in requirements | Robot Framework |
| JS/TS | `jest` in package.json | Jest |
| JS/TS | `vitest` in package.json | Vitest |
| JS/TS | `mocha` in package.json | Mocha |
| Java | `junit` in pom.xml/build.gradle | JUnit 4/5 |
| Java | `testng` in pom.xml | TestNG |
| C# | `NUnit` / `xunit` in csproj | NUnit / xUnit |

#### 第三步：检测测试文件位置

| 语言 | 测试文件模式 | 典型位置 |
|------|-------------|----------|
| Go | `*_test.go` | 与源文件同目录 |
| Python | `test_*.py` / `*_test.py` | `tests/` 或与源文件同目录 |
| JS/TS | `*.test.ts` / `*.spec.ts` | `__tests__/` 或 `*.test.*` |
| Java | `*Test.java` | `src/test/java/` |
| C# | `*Tests.cs` | 对应测试项目 |
| Rust | `#[test]` in `*.rs` | 同文件或 `tests/` |
| Robot | `*.robot` | 配置指定 |

### 适配策略

识别完成后，按以下维度适配：

- **命名规范** — 使用目标语言/框架的惯例
- **断言库** — 使用项目已有的断言库（不要引入新的）
- **目录结构** — 测试文件放在项目约定位置
- **构建标签** — 如 Go 的 build tags
- **配置文件** — 如 pytest.ini, jest.config.ts

## 测试源扫描策略

### 源类型识别

测试用例可能存在于以下位置：

#### A. Git 仓库（代码内）

**扫描方法**：
```
1. 搜索测试文件模式（*_test.go, test_*.py, *.test.ts 等）
2. 分析测试文件内容，提取函数/类/用例列表
3. 识别测试框架和断言模式
4. 统计覆盖范围
```

**关键问题**：
- 有多少测试文件？多少个测试函数/用例？
- 覆盖了哪些模块？哪些模块没有测试？
- 测试之间是否有依赖？（应该独立）
- 有没有跳过的测试？（`t.Skip`, `@skip`, `pytest.mark.skip`）

#### B. Git 仓库（配置文件驱动）

**典型场景**：JSON/YAML/CSV 驱动的测试用例

**扫描方法**：
```
1. 识别测试用例文件（cases/*.json, testdata/*.yaml 等）
2. 解析 Schema 或文件结构
3. 列出所有用例及其覆盖范围
4. 检查与源代码的对应关系
```

**本项目示例**：
- `rain-qa-func/cases/fight_cases/*.json` — 战斗测试用例
- `rain-qa-func/cases/excel_cases/` — Excel 检查规则
- `rain-excel-checker/xlsx/tests/resources/` — 测试资源

#### C. 飞书云文档

**扫描方法**：

1. 使用 `mcp__feishu-mcp__search-doc` 搜索测试相关文档
   - 关键词："测试计划"、"测试用例"、"回归测试"、"冒烟测试"
2. 使用 `mcp__feishu-mcp__fetch-doc` 获取文档内容
3. 解析文档中的测试用例结构（表格、列表、检查项）
4. 提取可操作的测试步骤

**文档结构识别**：
- 表格形式：识别列头（步骤、操作、预期结果）
- 列表形式：识别层级（模块 > 功能 > 用例）
- 检查项：识别 `- [ ]` / `- [x]` 格式

#### D. 飞书多维表格

**扫描方法**：

1. 使用 `mcp__feishu-mcp__list-docs` 浏览知识库
2. 定位测试管理相关的多维表格
3. 读取表格结构和数据
4. 分析测试用例状态、覆盖率、执行结果

**关注字段**：
- 用例状态（待执行/通过/失败/阻塞）
- 优先级（P0/P1/P2/P3）
- 模块归属
- 最近执行时间
- Bug 关联

#### E. 其他系统

| 系统 | 集成方式 | 说明 |
|------|----------|------|
| Jira | API/导出 | 测试用例作为 Issue 管理 |
| TestRail | API | 专业测试管理工具 |
| Excel 文件 | 直接解析 | 离线测试用例管理 |
| Wiki (Confluence) | API/导出 | 文档形式管理 |
| 自研系统 | 具体分析 | 需要了解接口 |

### 通用扫描流程

```
1. 识别可访问的测试源
   → Git 仓库？飞书文档？本地文件？在线系统？

2. 逐源扫描
   → 代码测试：搜索测试文件 + 分析覆盖
   → 文档测试：搜索关键词 + 解析结构
   → 配置测试：定位用例文件 + 解析格式

3. 汇总到统一视图
   → 各源的测试数量、覆盖范围、质量评估
```

## 测试盘点评估

### 盘点维度

对发现的测试进行以下维度评估：

#### 1. 覆盖度评估

| 维度 | 评估方法 | 理想状态 |
|------|----------|----------|
| 模块覆盖 | 测试文件 vs 源文件对应关系 | 核心模块 100% |
| 分支覆盖 | 测试是否覆盖 if/switch 分支 | 核心逻辑 >80% |
| 功能覆盖 | 测试场景 vs 需求文档对照 | P0 功能 100% |
| 接口覆盖 | API 端点 vs 测试用例对照 | 公开接口 100% |

#### 2. 质量评估

| 维度 | 检查项 | 常见问题 |
|------|--------|----------|
| 有效性 | 测试是否真正验证了逻辑 | 欺骗性测试（assert.True(true)） |
| 独立性 | 测试是否可独立运行 | 测试间依赖、共享状态 |
| 可维护性 | 测试是否容易理解和修改 | 硬编码、魔数、过长 setup |
| 稳定性 | 测试是否稳定通过 | Flaky test、依赖时序/网络 |
| 时效性 | 测试是否与代码同步 | 过时的断言、废弃的接口调用 |

#### 3. 分类统计

按类型统计现有测试：

| 分类 | 示例 | 统计维度 |
|------|------|----------|
| 按类型 | 单元/集成/端到端 | 各类型数量和占比 |
| 按优先级 | P0/P1/P2/P3 | 各级别覆盖情况 |
| 按状态 | 通过/失败/跳过/未执行 | 健康度 |
| 按所有者 | 开发/QA/自动化 | 维护责任分布 |
| 按时效 | 本周新增/长期未执行 | 活跃度 |

## 测试优化指导

### 常见问题与解决方案

#### 覆盖率不足

**症状**：核心模块缺少测试

**策略**：
1. 优先补充核心路径的冒烟测试（快速提升安全网）
2. 对高风险模块逐个补充单元测试
3. 对关键接口补充集成测试
4. 不要追求一次性 100%，分层推进

#### 测试质量低

**症状**：测试通过但 bug 依然出现

**策略**：
1. 检查断言有效性（是否真正验证了结果）
2. 补充边界条件测试
3. 补充异常路径测试
4. 检查 Mock 是否过度简化了真实行为

#### Flaky 测试

**症状**：测试偶尔失败

**策略**：
1. 识别 flaky 测试（连续运行 10 次观察通过率）
2. 分析失败原因（时序/并发/外部依赖/数据竞争）
3. 隔离修复（增加确定性等待、消除共享状态）
4. 如果无法修复，降级为手动测试并标记原因

#### 测试维护成本高

**症状**：修改代码后大量测试需要更新

**策略**：
1. 检查是否过度 Mock（Mock 了不需要 Mock 的东西）
2. 重构测试结构（公共 setup/teardown、测试数据工厂）
3. 降低测试与实现的耦合（通过接口而非实现细节测试）
4. 考虑是否某些测试价值不足以维护

### 优化优先级排序

```
1. P0 功能无测试     → 立即补充（高风险）
2. Bug 回归测试缺失  → 立即补充（已知风险）
3. 核心逻辑覆盖不足  → 短期补充（1-2 周）
4. Flaky 测试修复    → 短期修复（影响信任度）
5. 测试结构优化      → 中期改进（降低维护成本）
6. 覆盖率提升        → 长期目标（持续推进）
7. 测试文档完善      → 按需进行
```

### 迭代节奏建议

**每次迭代回答**：
1. 上一轮补充的测试有效性如何？（发现了几个 bug？）
2. 现在最大的覆盖缺口在哪？
3. 有没有新增的 flaky 测试？
4. 下一步优先补充什么？
