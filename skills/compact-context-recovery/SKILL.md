# Compact 后上下文恢复

> `/compact` 压缩对话后，之前的计划执行上下文和角色认知会丢失。此技能定义恢复流程，确保 AI 在压缩后仍能追踪计划进度并保持调度者身份。

## 为什么需要这个技能

compact 操作会：
- 将完整对话压缩为摘要，丢失细节和"正在做什么"的连续感
- 保留任务列表（TaskList）但丢失任务的上下文（为什么创建、之前遇到过什么问题）
- 保留计划文件（`.claude/plans/`）但丢失"已执行到哪个阶段"的认知
- **丢失角色认知** — compact 前是调度者（分配 subagent），compact 后变成执行者（亲自改代码）
- 导致 AI 脱离计划自行行动，或重复已完成的工作

## 数据来源：progress-tracker 技能

进度数据的写入由 **progress-tracker** 技能负责，本技能只负责读取和恢复。

progress-tracker 在开发过程中持续维护 `memory/active-plan.md` 文件，包含：
- 每个 Phase 的完成状态（✅⏳🚫🔲）
- 当前角色（调度者/执行者）
- 活跃 agent 列表
- 下一步操作
- 变更日志

## 恢复流程

检测到 compact/压缩/会话恢复信号时，**在任何其他操作之前**执行以下步骤：

### 第 1 步：扫描现有上下文

按以下优先级读取数据源：

```
1. 读取进度文件：memory/active-plan.md（progress-tracker 维护，最可靠）
2. 读取计划文件：.claude/plans/ 目录下的 .md 文件
3. 读取任务列表：调用 TaskList 获取当前任务状态
4. 读取 memory 索引：memory/MEMORY.md 确认是否有进度记录条目
```

### 第 2 步：对比计划与任务状态

数据源优先级：**进度文件 > TaskList > 摘要推断**

- 如果 `memory/active-plan.md` 存在，直接解析其中的 Phase 状态、角色、活跃 agent（最可靠）
- 如果进度文件不存在，回退到 TaskList 和摘要推断（需要更谨慎确认）

### 第 3 步：恢复角色认知

检查 memory 中记录的角色：
- 如果之前是**调度者**，恢复调度者模式：不亲自修改代码，通过 subagent 并行执行
- 如果之前是**执行者**，继续执行模式
- 如果角色信息缺失，根据任务复杂度判断：涉及多文件修改时默认调度者

### 第 4 步：向用户确认进度

**必须**向用户输出类似以下格式的进度报告，等待用户确认：

```
📋 计划恢复报告

计划文件: [计划名称]
├── Phase 1: [名称] — ✅ 已完成
├── Phase 2: [名称] — ✅ 已完成
├── Phase 3: [名称] — ⏳ 进行中（任务 #X）
├── Phase 4: [名称] — 🔲 未开始
└── Phase 5: [名称] — 🔲 未开始

角色: 调度者（subagent 并行开发）
当前待办任务:
- #X [in_progress] 任务描述
- #Y [pending] 任务描述

请确认：以上进度是否准确？我应该从哪里继续？
```

### 第 5 步：继续执行

用户确认后，从确认的阶段继续执行计划。不要跳过未确认的阶段。

## 触发信号

以下信号出现时必须触发恢复流程：

| 信号 | 检测方式 |
|------|----------|
| 对话摘要标记 | 系统提示中出现 "This session is being continued from a previous conversation" 或 "Summary:" |
| compact 命令 | 用户执行了 `/compact` |
| 用户主动提及 | 用户说"compact后"、"压缩后"、"继续之前的工作"、"接着做" |
| 存在计划文件 | 系统提示中加载了 `.claude/plans/` 下的计划文件 |
| 任务列表有遗留 | TaskList 中存在 in_progress 或 pending 状态的任务 |
| 用户追问 bug | 用户报告了之前对话中的 bug，说明上下文可能丢失 |

## 恢复后的行为规范

恢复上下文后，遵守以下规范防止再次丢失：

1. **Phase 完成后使用 progress-tracker 更新进度** — 调用 PHASE_DONE 写入 `memory/active-plan.md`
2. **遇到阻塞使用 progress-tracker 记录** — 调用 PHASE_BLOCKED 写入阻塞原因
3. **计划有变更时，更新计划文件** — 不要只在对话中讨论变更
4. **保持调度者身份** — 如果之前是调度者，compact 后不要变成执行者亲自改代码
5. **角色变更时通过 progress-tracker 记录** — PHASE_DONE 时附带记录当前角色

## 边界情况

### 没有计划文件
如果 `.claude/plans/` 为空或不存在，说明之前没有活跃的计划。跳过恢复流程，正常响应用户。

### 没有 memory 进度记录
如果 memory 中没有进度记录（`memory/active-plan.md` 不存在），说明 progress-tracker 未被使用（旧版会话或执行纪律未遵守）。回退到 TaskList 和摘要推断进度，但需要更谨慎地向用户确认。

### 计划文件内容与对话无关
如果计划文件的主题与用户当前请求完全无关（例如旧会话的计划），询问用户是否需要清理旧计划。

### 任务列表为空
如果 TaskList 为空但存在计划文件，说明任务已完成但计划未标记。向用户确认计划是否已全部完成。

### 多个计划文件
如果存在多个计划文件，优先关注最近修改的那个。如果多个计划都未完成，向用户确认应该继续哪个。

## 不要做的事

- ❌ 假设之前做到哪里了，直接继续执行
- ❌ 忽略计划文件，当作没有计划存在
- ❌ 把摘要当作完整上下文，遗漏摘要中未记录的细节
- ❌ 在未向用户确认的情况下跳过阶段
- ❌ compact 后假装什么都没发生
- ❌ compact 后丢失调度者身份，亲自改代码消耗上下文
- ❌ 只依赖对话摘要恢复进度，不读取 memory 进度文件
